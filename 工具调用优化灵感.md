# 工具调用优化灵感

## 缘起

分析 `DSPlus/analysis_logs/2026-06-24.jsonl` 中记录的 Claude Code 开发行为日志（308 条请求日志，全部来自 DeepSeek 模型接入 Claude Code 等开发应用时产生的真实流量），发现工具调用的设计模式值得深入讨论。

## 发现：读可以并行，改必须串行

在 **262 次**含有 tool_calls 的 API 调用中：

| 模式 | 次数 |
|------|------|
| 单次 1 个工具 | 213 |
| 单次 2 个工具（如 bash+bash, read+glob） | 30 |
| 单次 3 个工具 | 14 |
| 单次 4 个工具 | 5 |
| **同一次含多个 edit** | **0** |
| **edit 与其他工具混搭** | **0** |

**多工具调用是一次性生成的，还是多次请求生成的？**

是在同一次 API 响应中一次性生成的。每条日志的 finish_reason 都是 "tool_calls"。例如 `sess_djgk8zcy5gxo` 第 1 轮模型返回了两个 bash 调用（git status + git log），出现在同一次模型输出中，而非分两次请求。

**工具结果返回方式：合并后一起返回。**

所有工具结果在同一次请求中以 tool role 消息批量提交。模型同时看到所有结果，再决定下一步输出。

**但当涉及 edit 工具时，永远是一次一个，绝不批量。**

每个 edit 独占一轮 API 请求/响应周期。流程是：

```
read → edit → edit → edit → read → edit → edit → bash(语法检查) → stop
```

每次 edit 调用：
1. 模型生成 **1 个** edit 调用
2. 客户端执行编辑，返回结果
3. 模型看到结果后，决定下一步（下一个 edit 或 read 或 bash）
4. 重复，直到完成

本质上是：**工具的运行肯定是一次一个的这样改。但我想知道是大模型返回结果时，它是返回多个还是等工具改完一个，它再去给下一个修改？这涉及到了，到底是多次调用还是单次调用大模型？**

结论是：**每个 edit 都是单独一次大模型调用**。不是"一次返回多个 edit 再一个个执行"，而是：

```
LLM 调用 #1 → 返回 1 个 edit → 执行 → 结果喂回
LLM 调用 #2 → 返回 1 个 edit → 执行 → 结果喂回
LLM 调用 #3 → 返回 1 个 edit → 执行 → 结果喂回
...
```

以 `sess_djgyvtoel0g8` 为例（17 轮 API 调用做了 9 个 edit）：

| 轮次 | 返回内容 | finish_reason |
|------|----------|--------------|
| 2 | edit(return +input_box=None) | tool_calls |
| 3 | edit(空列表 +input_box=None) | tool_calls |
| 4 | edit(空 big +input_box=None) | tool_calls |
| 5 | edit(input_rect 逻辑) | tool_calls |
| 7 | edit(keep/drop + input_rect) | tool_calls |
| 9 | edit(analyze 返回 +input_box) | tool_calls |
| 10 | "Now update PreviewData..." + edit | tool_calls |
| 11 | edit(input_box = result.get) | tool_calls |
| 12 | edit(input_box 绘制代码) | tool_calls |
| 14 | edit(PreviewData 构造 +input_box) | tool_calls |

每次 last_content_len=0（纯工具调用不返回文本），chat_entries=2（只有 tool result + 新 assistant），每个 edit 独占一次 LLM 调用。

## 为什么要这样设计？社区的选择

目前社区存在两种设计：

### 顺序模式（Claude Code 等）

模型一次输出一个 edit → 执行 → 结果回来 → 再输出下一个。

设计意图推断：
- read/bash/glob 是幂等、可并行的，可以一次性发起多个，一起执行，一起返回
- edit 有顺序依赖，后一个编辑可能影响前一个的结果（代码上下文变更），需要看到每个编辑的结果后再决定下一步
- 每个 edit 需要精确的 old_string → new_string 匹配，如果并发编辑同一文件的不同部分，可能互相干扰

### 批处理模式（Aider）

Aider 走了批处理的路——模型一次性输出所有改动，脚本批量应用。它的 edit_format 支持 whole/diff/udiff 三种格式，模型只需在一个响应里返回多个 SEARCH/REPLACE 块。

Aider 解析出所有 edits，挨个 apply（wholefile_coder.py:apply_edits 循环处理），然后一次性返回结果给模型（或是通过 git 做原子回滚）。这样一次 LLM 调用 = N 个文件修改。

### 两种设计的对比

| | 顺序（Claude Code） | 批处理（Aider） |
|---|---|---|
| 每次 LLM 调用 | 1 个 edit | N 个 edit |
| 可靠性 | 高 —— 看到结果再决定下一步 | 低 —— 一个 edit 失败不感知其他 edit |
| 冲突风险 | 无 —— 始终基于最新文件状态 | 有 —— 同文件区域重叠会冲突 |
| 成本 | N 次调用 × 上下文重传 | 1 次调用 |

Claude Code 的设计师选择了"宁可慢，不可错"：每个 edit 都是基于实时文件状态的精确字符串替换，old_string 匹配失败就报错，必须等模型修正后重试。这在 edit 的 read-before-edit 校验机制下尤其重要——两次编辑同一文件的不同行，第一次改了行号偏移后，第二次的 old_string 可能就不匹配了。

Aider 则赌"模型一次生成足够精确"，配合 git 回滚兜底。代价是实践中经常出现 Search/Replace 失败需要重来。

## 新的思路：现在的 LLM 可能已经可以批处理了

其实现在大模型的遵循度已经很强了。最初我觉得这样设计没问题，但如今也许已经到了可以批处理的时候。大模型给出的是批处理，但我们的工具当然还是一次一次的处理。当遇到问题的时候，可以把冲突返回给一个子 agent，这个子 agent 专门用来解决写入冲突。子 agent 就可以一次一次的写入，并且子 agent 的写入没有主 agent 的超长上下文，应该能够大幅节约 token。如果子 agent 的写入也多次失败，再报告给主 agent。

### 这个设计为什么合理

**1. 大部分 edit 不会冲突**

从日志数据看，`sess_djgyyqy2guho` 的 12 个 edit：
- Turn 2-4: 改 preview_app.py（同一函数不同行）—— 有冲突风险
- Turn 5-6: 改 region_overlay.py —— 不同文件，零冲突
- Turn 7: 改 preview_app.py —— 跨文件了
- Turn 9-13: 改 preview_app.py —— 同一文件不同函数，大部分不重叠

实际冲突率远低于 100%。批处理赌的是大部分 edit 独立，少数冲突由子 agent 兜底。

**2. 子 agent 上下文极小**

主 agent 的上下文装了一堆：项目结构分析、文件读取、调试输出、用户意图。子 agent 只需要：
- 冲突文件当前内容
- 主 agent 对该文件的原始 edit 指令（old_string → new_string）
- 上一轮冲突尝试的报错信息

相比主 agent 的数万甚至数十万 token，子 agent 可能只要几千。

**3. 退路清晰**

- 子 agent 成功了 → 零额外 LLM 调用，完美
- 子 agent 也卡住（比如 old_string 被前面的 edit 改了）→ 报告主 agent，主 agent 知道整体意图，可以重新规划 edit

### 实际可行的架构

```
主 LLM 输出:
  edit("fileA", oldA → newA)
  edit("fileB", oldB → newB)
  edit("fileC", oldC → newC)

工具层: 逐个尝试，记录结果
  ✓ fileA: OK
  ✗ fileB: old_string 不匹配 (因 fileA 改了同一区域的上下文)
  ✓ fileC: OK

有失败? → 子 agent(指令: "fileB 要将 {意图描述} 改为新逻辑，
          当前文件内容: {fileB 实时内容}，请生成正确的 edit")
  → 子 agent → edit(fileB) → OK → 报告成功
  或
  → 子 agent 重试 3 次失败 → 报告主 agent 协助
```

### 分层回退

本质是把 3 层回退引入了编辑流程：

1. **最优路径**：全批处理成功 —— 一次 LLM 调用解决 N 个文件修改
2. **中间路径**：子 agent 解冲突 —— 轻量级子 agent 使用极小上下文逐个解决冲突
3. **最差路径**：回主 agent 重规划 —— 主 agent 利用对全局意图的理解重新设计编辑方案

每一层都比前一层更贵但也更强。

### 注意事项

- **冲突检测粒度**：两个 edit 改不同文件肯定无冲突。改同一文件的非重叠行也可能无冲突（加的 import 不会影响下面的 old_string）。真正的冲突是两个 edit 改了同一个 old_string 的邻接区域。这个可以按文件分组，同文件的有序重试即可。

- **子 agent 的工具权限**：只需要 Read + Edit 两个工具，不需要 Bash/WebFetch，权限可以缩到最小。

- **主 agent 的输出格式**：需要让主 agent 知道它会以 batch 模式输出，因此每个 edit 要独立可执行，不能依赖前一个 edit 的结果。
