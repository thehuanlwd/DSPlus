# 每日分析日志自包含哈希去重设计规范 (Log Deduplication Design)

本规范定义了 `analysis_logs` 中系统提示词（System Prompt）及大段重复文本在持久化过程中的去重设计规则，用以优化磁盘存储空间并提升 I/O 效率。

---

## 1. 核心去重策略：单日日志自包含字典 (Self-Contained Daily Dictionary)

为了避免维护全局大字典所带来的文件损坏、同步锁竞争以及备份迁移困难等问题，本方案采用**“单日日志自包含字典”**策略：

* 每个日期的日志文件 `YYYY-MM-DD.jsonl` 是一个完全独立、自包含的数据实体。
* 日志文件中的每一行仍然保持统一的 JSON 格式（反序列化为 `TraceEvent`）。
* 在同一天内，若某个系统提示词（`role == "system"` 且长度 `> 500` 字节）首次出现，在当前事件行中保持明文写入，并作为该日日志的“字典定义项”存在。
* 随后出现的相同系统提示词，在事件行中仅记录其哈希引用 `"$ref_dict:<hash>"`。
* 跨天写入时，新一天的日志文件会重新写入一次该提示词的明文，新一天内后续请求继续去重。

---

## 2. 详细实现规则

### 2.1 内存状态维护
在 `AnalysisService` 中维护一个仅在写盘协程中访问的非并发安全 Map：
```go
writtenHashes map[string]bool
```
* **生命周期**：每次程序初始化、跨天生成新日志文件时清空该 Map。
* **重启同步**：系统重启时，通过顺序加载今日已写入日志（`loadHistoryFromDisk`），在还原会话的同时，重新将已扫描到的明文系统提示词哈希收集回 `writtenHashes`，实现热重载同步。

### 2.2 计算规则
* **目标筛选**：仅对 `ChatMessage.Role == "system"` 且 `len(ChatMessage.Content) > 500` 字节的文本进行去重计算。
* **哈希算法**：采用高效安全的 SHA-256 算法，并截取前 16 位字符（或直接使用 32 位 MD5），形如 `sha256_xxxxxx`。

### 2.3 序列化写入行为 (writeToDisk)
当 `AnalysisService` 将事件写入磁盘时，执行以下转换逻辑：
1. 复制事件副本，防止修改内存中正在被 API 使用的数据。
2. 遍历事件副本中的 `ChatHistory` 数组。
3. 若发现符合筛选条件的 `system` 消息：
   - 计算其内容的哈希值。
   - 若哈希值**不在** `writtenHashes` 中：
     - 保留原始明文，继续序列化并写入日志。
     - 将哈希值加入 `writtenHashes` 中。
   - 若哈希值**已存在**于 `writtenHashes` 中：
     - 将消息的 `Content` 改写为 `"$ref_dict:<hash>"`（例如：`"$ref_dict:md5_e10adc3949ba59abbe56e057f20f883e"`）。
     - 写入序列化后的 JSON 行。

### 2.4 反序列化读取行为 (loadHistoryFromDisk / API)
当系统需要从磁盘加载日志以恢复内存缓存，或在前端导出 Markdown 报告时，执行以下回填逻辑：
1. 在读取特定日期日志的解析上下文中，声明一个临时的字典 Map：`localDict := make(map[string]string)`。
2. 逐行读取 `.jsonl` 文件并进行 `json.Unmarshal`。
3. 检查读取到的 `ChatHistory` 消息：
   - 若 `Role == "system"` 且内容**不是**以 `"$ref_dict:"` 开头，计算哈希并存入：
     ```go
     localDict[hash] = msg.Content
     ```
   - 若 `Role == "system"` 且内容**是**以 `"$ref_dict:<hash>"` 形式存在：
     - 从 `localDict` 中查找对应的原始文本。
     - 若找到，则将 `Content` 替换还原为原始文本；若未找到（理论上不应发生，除非文件发生非顺序性损毁），则保留 `"$ref_dict:<hash>"`。

---

## 3. 兼容性与鲁棒性保证

### 3.1 向后兼容性 (Backward Compatibility)
对于本功能上线前已经生成的历史明文 `.jsonl` 日志，由于其 `system` 消息内容本身不包含 `"$ref_dict:"` 前缀，反序列化加载时会自动识别为明文，存入临时字典并正常加载，完全向后兼容。

### 3.2 文件损毁容错 (Fault Tolerance)
如果日志文件因为断电等异常造成物理损坏：
* 若损坏部分在文件末尾，不影响前半部分已加载的明文字典项与引用行。
* 若损坏部分恰好包含了首发的明文行，后续引用行在 `localDict` 中找不到对应哈希时，会优雅回退保留 `"$ref_dict:<hash>"` 字符串，而不会导致程序崩溃或解析中断。

---

## 4. 性能估算

* **CPU 开销**：计算一次 30 KB 系统提示词的哈希仅需约 100 微秒，且仅在每个会话的首轮发生，CPU 负载极低。
* **I/O 节约**：将每次会话的写入量从 30 KB+ 降至约 100 字节，大幅减少物理磁盘写入，降低磁盘写放大效应。
* **网络传输**：API 层在向前端传输前，可以在后端统一回填数据，也可选择向前端传输去重后的精简 JSON 并在前端缓存字典做渲染，降低网络带宽开销。
